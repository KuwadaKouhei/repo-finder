# 技術スタック

> 本アプリで採用する技術と、その選定理由・代替案をまとめた参照ドキュメント。
> スタックの変更時は本書を更新する。
> 本書はAIを用いて今回の要件に合う技術スタックを複数調査し、開発者が選定することで決定したものを載せている。
> 不採用理由などはその技術スタックを調べたうえで決定したことである。
> 関連: DESIGN.md（基本設計）/ REQUIREMENTS.md / DETAIL_DESIGN.md / GITHUB_API.md

## 選定の自由度（3層）

技術スタックは「選定の自由度」で3層に分かれる。混同しないよう分けて扱う。

| 層 | 内容 | 扱い |
| --- | --- | --- |
| 課題で固定 | Next.js 16+ / App Router / テスト記述 | 制約。選定の余地なし |
| 実質固定 | 言語（TypeScript） | 事実上の標準。理由は述べる |
| 自分で選ぶ | テスト・スタイリング・UI・補助ライブラリ | 本来の技術選定。理由が評価対象 |

## バージョン(AIを用いてLatestVersionを調査)

開発時点の最新安定版を基準に固定する。下表を `package.json` / `.nvmrc` / `engines` の基準とする。

| 対象 | 確定バージョン | 補足 |
| --- | --- | --- |
| Node.js | **22.21.0**（最低 20.9+） | Next.js 16 は Node 20.9 以上が必須。使用環境がv22.21.0であるためそれを採用。 |
| パッケージマネージャ | **npm** | Node.js同梱で追加導入が不要。 |
| Next.js | **16.2 系**（例 `16.2.7`） | App Router |
| React / React DOM | **19.2 系** | Next 16 が前提とする世代。`@types/react`・`@types/react-dom` も最新に揃える |
| TypeScript | **5.x**（最低 5.1+） | `strict: true`。Next 16 の要件は 5.1 以上 |
| Tailwind CSS | **4.x**（例 `4.2`） | v4 系。Viteプラグイン/CSS-firstの設定 |
| shadcn/ui | **CLI 最新で取り込み** | コードを所有する方式。バージョンは「取り込み時点のコンポーネントコード」で固定（コミットで担保） |
| Radix UI | shadcn が引き込む版に追従 | 直接は固定しない |
| next-themes | **0.4 系** | ダークモード（FOUC・システム追従・localStorage） |
| Vitest | **4.x**（例 `4.1`） | Vite 6+ / Node 20+ を要求。Vite は Vitest が引き込む |
| @testing-library/react | **16.x** | React 19 対応版 |
| @testing-library/jest-dom | 最新 | カスタムマッチャ |
| MSW | **2.x** | ネットワーク境界モック |

### バージョン方針

- **固定の粒度**: アプリの根幹（Next・React・TypeScript）はメジャー＋マイナーを意識して固定。ライブラリはキャレット（`^`）で当該メジャー内の更新を許容。ロックファイルで実バージョンを固定する。
- **Node のLTS方針**: 偶数（LTS）系のみを使う。`.nvmrc` で開発環境を揃え、`engines` で逸脱を弾く。
- **ブラウザ下限**: Next.js 16 の対応（Chrome/Edge/Firefox/Safari 16.4 以上）に準拠（REQUIREMENTS.md §4.6 / NFR-C1 と整合）。
- **更新の扱い**: 提出時点の最新安定版で固定する。提出後の追従はしない（課題のため）。実務では Renovate/Dependabot で定期更新するが今回はスコープ外。

## 各選定の補足

### 言語（TypeScript）

Next.jsで書く以上、事実上の標準。「選定」というより前提だが、型安全・アダプタ層（ACL）の契約を型で表現できる点を理由として明示する。`strict: true` を有効にする。

### テスト（Vitest + RTL + MSW）

詳細設計・設計思想（テスティングトロフィー）と整合。アダプタは純粋関数として単体テスト、APIエラー写像と検索フローはMSWで境界モックする。

### スタイリング（Tailwind）と UIコンポーネント（shadcn/ui）

- スタイリングは Tailwind CSS。
- UIは **shadcn/ui を採用し、コードを所有して自分で改変・拡張する**（「shadcnを土台に自力で組み立てる」）。採用理由は、アクセシビリティ（Radixベースでキーボード操作・フォーカス・ARIA が確保される）、ダークモード（CSS変数テーマで next-themes と素直に噛み合う）、コードを所有できる方式であること。
- **モックとの関係**: モックは見た目の参照のみとし、コードは流用しない。モックのデザイントークン（アクセントカラー・角丸・余白スケール）を shadcn のテーマ変数にマッピングし、モック固有のアニメーション（カウントアップ・アクセントエッジ・float-in 等）は shadcn に含まれないため自分で追加する。すなわち **shadcnは構造とa11yの土台を提供し、デザインの個性（トークン・アニメーション）は自力で乗せる**という分担。
- コンポーネント→shadcn の割り当ては DETAIL_DESIGN.md を参照。

### ダークモード（next-themes）

- SSRでは初回描画でテーマがちらつく（FOUC）。next-themes が FOUC・システム設定追従・localStorage永続を正しく解決するため、ダークモードのためにこの依存を入れるのは正当な例外とする（「原則追加しない」方針の明示的な例外）。
- テーマは個人の表示設定のため localStorage に保持（検索条件＝URL、テーマ＝localStorage の使い分け）。

## 採用しなかった技術（検討と見送り）

検討したが採用しなかったもの。「比較した上で選んでいる」ことを示すための記録。

### 言語・フレームワーク周辺

| 技術 | 何の候補か | 見送り理由 |
| --- | --- | --- |
| JavaScript | 言語 | 型がなく、アダプタ層の契約を表現できない。プロダクション想定に合わない |
| Pages Router | Next.jsのルーティング | RSC・サーバーコンポーネントを活用できず、トークン秘匿・初期JS削減・サーバーフェッチ前提の設計と噛み合わない。App Routerを採用 |

### データ取得・状態管理

| 技術 | 何の候補か | 見送り理由 |
| --- | --- | --- |
| TanStack Query | クライアントデータ取得・キャッシュ | 取得はRSCで完結。トークン秘匿・初期JS削減を優先。無限スクロールやミューテーション主体なら再検討の余地 |
| Redux / Zustand / Jotai | グローバル状態管理 | 状態はサーバー=RSC・共有=URL・個人/一時=useState/localStorageで充足。共有すべき複雑なクライアント状態が存在せずオーバースペック |
| SWR | データ取得 | 同上（RSCで足りる） |
| Route Handler 経由のクライアントfetch | データ取得経路 | サーバーフェッチで足り、経路を増やす必要がない。トークン秘匿の観点でもサーバー取得が素直 |

### スタイリング・UI

| 技術 | 何の候補か | 見送り理由 |
| --- | --- | --- |
| CSS Modules / プレーンCSS | スタイリング | フロント特化選考でCSS実装力を見せたい場合は有力だったが、限られた時間で整った見た目を作るためTailwindを優先 |
| MUI / Chakra UI 等 | UIライブラリ（完成品） | デザイン上書きが重く、独自デザインと相性が悪い。ロックインも避けたい |
| 全部自前のUI | UIコンポーネント | a11y（キーボード・フォーカス・ARIA）を正しく作る手間が大きい。今回は提出期限の制約もあるため不採用 |
| Radixを直接利用（shadcnなし） | ヘッドレスUI | shadcnがRadixを土台にしつつコード所有・改変の利便を足すため、shadcn採用で包含される |

### テスト

| 技術 | 何の候補か | 見送り理由 |
| --- | --- | --- |
| Jest | テストランナー | ESM対応・実行速度でVitestに劣る。Vite/Next環境ではVitestが素直 |
| fetch直モック | テストのモック手法 | 実装に結合し本番と乖離する。ネットワーク境界モック（MSW）を採用 |
| Cypress | E2E | E2Eは本課題では任意。入れるならPlaywrightでハッピーパス1本に留める |

### その他

| 技術 | 何の候補か | 見送り理由 |
| --- | --- | --- |
| date-fns / dayjs | 日付整形 | 相対表示等は `Intl` で代替できる。依存ゼロを優先（必要になれば再検討） |
| 自前FOUC対策スクリプト | ダークモード | 自前でも可能だが、FOUC・システム追従・ハイドレーション整合を正しく書く手間に対し next-themes が確実。唯一の依存例外として採用 |

> 共通の判断軸: 「RSC・URL・標準APIで充足するものに依存を足さない（YAGNI）」「依存を足すのは、自前実装の手間・リスクがライブラリの利点を明確に下回るときだけ（shadcn・next-themes）」。
