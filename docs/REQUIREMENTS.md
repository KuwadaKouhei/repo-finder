# Requirements

## 1. 環境

- Next.js v16以降を使用する。
- App Routerを使用する

## 2. 明示要件

- ユーザーが何かしらのキーワードを入力した時、Github APIを用いてリポジトリを検索する。
- 検索結果は一覧表示
- 特定の結果を選択した時、該当リポジトリの詳細ページを表示。
  - リポジトリ名
  - オーナーアイコン
  - プロジェクト言語
  - Star数
  - Watcher数
  - Fork数
  - Issue数
- テストコードを記述。
- プロダクションを想定した実装。
- AIを使用した場合は利用方法をREADMEにまとめる。

## 3. 機能要件

### 3.1 検索

- キーワード入力
- 検索実行(API実行)
  - submit方式で発火
  - 空白や空文字では実行しない
- 検索状態のURL同期(検索キーワードは URL クエリ ?q= に反映される。リロード・URL共有で同じ検索結果が再現される。)
- 検索結果のsort機能
  - 関連度(デフォルト)
  - Star数
  - Fork数
  - 更新日時

### 3.2 一覧表示

- 検索結果一覧表示(リスト表示・グリッド表示)
- 検索結果リスト表示とグリッド表示の切り替え
- 各カードの表示項目（オーナー / リポジトリ名 / 説明 / 言語ドット / Star・Fork・Issue 数。グリッドはトピックも表示）
- ページネーション
- 詳細ページ遷移

### 3.3 詳細表示

- ページ実装
- 詳細項目表示（ リポジトリ名 オーナーアイコン プロジェクト言語 Star数 Watcher数 Fork数 Issue数）
- 追加表示項目（デザイン刷新で追加。いずれも `getRepository` の単一レスポンスから取得）
  - トピック（topics）
  - ライセンス（license.spdx_id。未設定/NOASSERTION は非表示）
  - 最終更新日（pushed_at。無ければ updated_at）
- 言語がnullの場合でも表示を崩さない
- Githubページへの外部リンク

### 3.4 状態表示(UI)

- 初期状態(未検索時は検索結果0件とは区別し、検索を促す初期表示を行う)
- ローディング表示
- 検索結果0件表示
- エラー状態
- 404表示(存在しないリポジトリの詳細URLにアクセスした時など)

### 3.5 テーマ切り替え(ダーク/ライトモード)

- ダークモードとライトモードを切り替えられる。
- 初期値はOSのシステム設定（prefers-color-scheme）に追従する。
- 選択したテーマは保持する（ブラウザを再訪しても維持。永続化先はlocalStorage）。
  - ※ テーマは共有対象ではない個人の表示設定のため、URLではなくlocalStorageに保持する（検索条件はURL、テーマはlocalStorage、という使い分け）。
- 初回表示でテーマがちらつかない（FOUC対策。詳細はDETAIL_DESIGN.md）。
- ライト/ダーク双方でコントラスト基準（§4.4）を満たす。

### 3.6 テスト

- アダプタ単体テスト
- クライアントエラーの写像
- 検索フロー結合テスト
- ソート統合テスト（sort/orderがクエリに反映・URL同期・変更時のpageリセット）
- 詳細描画テスト

### 3.7 ビジュアル / デザイン（Claude Design ハンドオフ準拠）

- 配色は zinc ニュートラル × ブルーアクセント固定。ライト/ダーク両対応。
- 一覧ページ上部にヒーロー（バッジ「EXPLORE OPEN SOURCE」／見出し「リポジトリを見つけよう」／1行サブコピー）。
- 共通ヘッダー（ロゴ＋サービス名＋テーマ切替）とフッターを全ルートで表示。
- アニメーション（いずれも `prefers-reduced-motion` で無効化）
  - カード入場（floatIn・stagger）／カードホバーのボーダースイープ
  - リスト⇄グリッド切替の FLIP
  - テーマ切替の円形リビール（View Transitions、未対応時は即時切替）
  - 件数・詳細スタットの数値カウントアップ
  - 検索促し／結果なしアイコンの浮遊
- 印刷・PDF化など凍結タイムライン環境では入場演出を無効化し、最終状態で表示する。

## 4. 非機能要件

### 4.1 性能効率

- 初期表示が速い(ServerComponentsで初期JSを最小化)
- 検索遅延を抑える
- 画像最適化(next/imageの使用)
- 不要な再取得を行わない（戻る操作で同じ検索を再実行しない。キャッシュする。鮮度が必要な検索は短命キャッシュもしくは非キャッシュ）

### 4.2 信頼性

- 障害時に死なない(API障害やレート制限で白画面にならず、フォールバックUIを表示する)
- エラー後にユーザー操作で再実行できる
- 競合の防止

### 4.3 保守性

- API変更の影響を局所化する(アンチコラプションレイヤ)
- リファクタ耐性のあるテスト
- 変更の凝集(機能単位ディレクトリで1つの機能変更を1つのフォルダにまとめる)
- フレームワーク非依存の部分は分離して、Next規約の結合を最小化する(ビジネスロジックなど)

### 4.4 アクセシビリティ

- WCAG 2.1 AA を目安とする
- キーボード操作で完結出来るようにする
- スクリーンリーダー対応
- レスポンシブ対応(モバイルでもデスクトップでも使えるようにする)
- ライト/ダーク双方でコントラスト基準を満たす

### 4.5 セキュリティ

- Githubトークンはサーバー環境変数に置く。
- 検索キーワードを適切にエンコードしてAPIへ渡す

### 4.6 互換性・移植性

- モダンブラウザ対応
- 移植容易性の向上(外界（GitHub API）との結合を lib/github に閉じ、データソース差し替え時の影響を局所化する。)

## 5. やらないこと(スコープ外)

YAGNIに基づき、やらないことを明記する

- 認証ログイン
今回はリポジトリの検索と表示・詳細表示がメインの為、ログインは不要。

- 無限スクロール
今回はページネーションを採用。無限スクロールは「戻る」時の位置復元やSSRの相性で複雑性が跳ね上がるため、コスト対効果が悪いと判断し不採用。
また、GithubAPIのレスポンスがページネーションと相性が良いため、今回はページネーションを採用。

- グローバルな状態管理(Reduxなど)
今回の要件としては状態管理はオーバースペックになるため不採用。
サーバー状態はRSC, 共有状態はURL, ローカル状態はuseStateで事足りる。
