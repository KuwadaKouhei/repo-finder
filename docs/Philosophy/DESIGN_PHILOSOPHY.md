# Design Philosophy

> 本プロジェクト全体の設計の拠り所。個別の判断（要件・画面・API）の上位にある「なぜそう作るか」を体系化する。
> UI に閉じた詳細は §8（および UI_DESIGN_PHILOSOPHY.md）に置く。
> 関連: REQUIREMENTS.md / DESIGN.md（基本設計）/ DETAIL_DESIGN.md / TECH_STACK.md / GITHUB_API.md / ADR

---

**変更容易性を最優先に、トレードオフを言語化して選ぶ。**

---

## 1. 価値観（最上位の優先順位）

判断に迷ったときの優先順位。上が優先。

1. **正しさ・安全性**（プロダクション想定。壊れない・秘密を漏らさない）
2. **変更容易性・保守性**（コードは書く時間より読まれ直される時間が長い）
3. **説明可能性**（なぜそうしたかを語れる。語れない技術選定はしない）
4. **シンプルさ**（YAGNI。必要になってから作る）
5. **見た目・利便性**（上位を満たした上で磨く）

---

## 2. アーキテクチャの思想

### 2.1 依存は内側（ドメイン）へ向ける

- Clean Architecture の Dependency Rule。`domain`（アプリ固有の概念）を中心に置き、UI もインフラもそこへ依存する。逆は禁止。
- フレームワーク（Next.js）は最も外側。ビジネスロジックは Next を import しない純粋な TS に保つ（フレームワーク非依存）。

### 2.2 外界はアダプタで隔離する（Anti-Corruption Layer）

- GitHub API という外部の都合（snake_case・巨大・`watchers_count` の罠）を、アダプタ（ACL）でドメイン型に変換し、UI へ侵入させない。
- 効果: API 変更の影響が `lib/github` に局所化される。データソース差し替えにも強い。

### 2.3 層を薄く保つ

- `app/`（Next 規約）は薄く、ルーティングと取得の起点に徹する。ロジックは `lib`/`domain` へ逃がす。
- 構造は規模に合わせて調整する（小さいうちは過剰な抽象化・体系化を避ける）。

---

## 3. 状態管理の思想

### 3.1 状態は3種に分類し、置き場所を選び分ける

| 種類 | 置き場所 | 例 |
| --- | --- | --- |
| サーバー状態 | RSC でサーバー取得（抱え込まない） | 検索結果・詳細 |
| URL 状態（共有可能） | URL クエリ | q / sort / order / page / view |
| クライアント状態（個人/一時的） | useState / localStorage | 入力中テキスト / テーマ |

### 3.2 帰結

- 共有・リロード・戻るで再現すべきものは URL に置く（URL as the single source of truth）。
- 個人の表示設定（テーマ）は localStorage。データのコピー（サーバー状態）を手で抱えない。
- グローバル状態管理（Redux 等）は、上記で充足するため不採用（YAGNI）。

---

## 4. データ取得の思想

- 取得はサーバー（RSC）で行い、トークンを秘匿する（クライアントへ出さない）。
- 初期 JS を小さく保ち、SEO・速度・安全性を取る。
- 鮮度と速度のトレードオフをキャッシュ方針で明示する（検索は鮮度優先、詳細は短命キャッシュ可）。

---

## 5. テストの思想

> 詳細は TEST_PHILOSOPHY.md。

- **テストを4層に分け、各層が「何を保証するか」で役割分担する**: ①ユニット＝最も壊れやすいロジック / ②コンポーネント＝ユーザーが見るUIの責務 / ③結合＝繋いだときの正しさ（部品同士の協調） / ④E2E＝実際のユーザー体験。
- 配分は**テスティングトロフィー**（静的解析 > ① > ③（最厚）> ④）。確信度/コストが最良なのは③結合テスト。
- **振る舞いをテストする**（実装詳細でなく role/label/text で検証）。壊れやすいテストは設計の悪さのシグナル。
- **ネットワーク境界でモック**（MSW）。本番に近いまま検証する。
- カバレッジは目的でなく、「壊れたら困る境界」（アダプタ＝契約）に厚く置く。
- テスト容易性を設計の制約として先に効かせる（表示は props のみの純粋な部品に）。

---

## 6. スコープの思想（やらないことを決める）

- **YAGNI**: まだ存在しない未来への投資（早すぎる抽象化）は負債。
- 「やらないこと」を理由付きで明記する。

---

## 7. セキュリティ・品質の思想

- トークンはサーバー環境変数（クライアント露出禁止）。
- 外部由来のデータ・URL は信頼しない（React 自動エスケープに委ね、`dangerouslySetInnerHTML` を使わない／URL はスキーム検証）。
- アクセシビリティは後付けでなく前提（WCAG 2.1 AA を目安、キーボード完結、ライト/ダーク両対応）。
- 異常系を必ず設計する（ローディング/空/エラー/404 の四状態＋初期）。「白画面で死なない」。

---

## 8. UI 設計の思想

> 詳細は UI_DESIGN_PHILOSOPHY.md。

- **コンポジション（合成）とコロケーションを第一原理**に置く（React / shadcn の根本思想と一貫）。機能単位で凝集させ、共有プリミティブだけを引き上げる。
- **プレゼンテーションとロジックを分離**する（表示は props を受け取る純粋な部品、取得・状態は上位の RSC）。
- UI は層の外殻。`domain`/`lib` は UI 設計思想の対象外（§2.1 と整合）。

---

## 9. 説明可能性の思想（メタ原則）

- すべての設計判断は「なぜそうしたか」「なぜ別案を採らないか」を語れる状態にする。
- 判断の経緯は ADR、現在の構成は各設計書、過程は Git 履歴に残す。
- 流行りの名前で採用しない。要件・規模・トレードオフから選んだ理由を一次の根拠とする。
