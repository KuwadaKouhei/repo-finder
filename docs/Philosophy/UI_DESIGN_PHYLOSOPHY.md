# UI Design Philosophy

> 本アプリのUI設計の拠り所。なぜこの方針を採るかを、規模・要件・トレードオフから記す。
> 関連: DETAIL_DESIGN.md（具体のディレクトリ・コンポーネント）/ TECH_STACK.md / REQUIREMENTS.md

---

## 1. 結論（一言）

**コンポジション（合成）とコロケーションを第一原理に置く。** 機能（feature）単位で凝集させ、複数機能で共有するプリミティブだけを共通の場所へ引き上げる。

---

## 1.1 変更履歴

UI設計は再利用性の高さからAtomic Designを最初採用しようとした。
しかし、今回のプロジェクトにおいてはオーバースペックであること。
他の設計思想でも再利用性の高さを実現出来ることから不採用とし、今回の設計思想であるコンポジション＋コロケーションを採用した。

## 2. 要件の特性（思想を選ぶ前提）

UI設計の方針は要件の規模・期間・体制に依存する。本アプリの特性は次の通り。

- 規模: 画面2つ（検索・詳細）、機能も限定的
- 期間: 短期（1〜2週間の課題）
- 体制: 単一開発者
- 技術: shadcn/ui を土台、Server Components 中心
- 品質: プロダクション想定（a11y・テスト・保守性が問われる）

この「小規模・短期・単一・合成志向」の組み合わせが、以降の判断の根拠になる。

---

## 3. 採用する3つの原則

### 3.1 コンポジション（合成）を第一原理に

- 小さく単純で「見た目だけ」を持つ部品を作り、組み合わせて複雑なものを作る。
- これは React と shadcn/ui の根本思想そのもの（例: `Card` / `CardHeader` / `CardContent`）。
- 利点: 再利用と拡張が自然にできる。組み合わせの自由を保てる。

### 3.2 コロケーション（関連するものを近くに置く）

- 1つの機能でしか使わないコンポーネント・型・テスト・スタイルは、その機能の近くに置く。
- 複数機能で共有するものだけを共通の場所（`components/ui` 等）へ引き上げる。
- 利点: 変更が一箇所に収まる（機能を直すときフォルダを横断しない）。保守性が上がる。

### 3.3 プレゼンテーションとロジックの分離

- 表示コンポーネントは props を受け取って描画するだけの純粋な部品にする。
- データ取得・状態は上位（本アプリでは RSC の `page`）が持つ。
- 利点: テスト容易性（props を渡すだけでテストできる）。Server Components 中心の構成と相性が良い。

---

## 4. 組織化の方針（機能軸＋共有の引き上げ）

- 画面・機能に固有のものは**機能（feature）の近くにコロケーション**する。
- 複数箇所で共有するプリミティブだけを共通の場所へ引き上げる:
  - `components/ui` … shadcn/ui（汎用プリミティブ）
  - `components/theme` … アプリ全体で使うテーマ関連
- 最初から共通化しない。2つ以上の機能で必要になってから引き上げる（必要になってから作る）。
- 具体のディレクトリは DETAIL_DESIGN.md を正とする。

---

## 5. アーキテクチャとの関係（UIは層の外殻）

- 本思想は UI（プレゼンテーション層）にのみ適用する。
- `domain/`（ドメイン型）・`lib/github/`（インフラ・ACL）は UI 設計思想の対象外（Clean Architecture の内側の層）。
- データは外側（`app/` の RSC）で取得し、内側のドメイン型に変換（ACL）してから、表示部品へ props で渡す。UI 層は API を直接叩かない。

---

## 6. アクセシビリティ・テストとの接続

- a11y は shadcn/ui（Radixベース）が土台を提供。自前部品は role/label/フォーカスを意識して合成する（要件 §4.4）。
- 表示とロジックの分離（§3.3）により、表示部品は props ベースで Testing Library から振る舞いをテストできる（要件 §3.6）。

---

## 7. 規模が変わったときの指針

- 機能が大幅に増える場合: 機能単位（feature）の粒度を保ちつつ、共有部品の整理を進める。
- 複数プロダクト/チームで UI を共有する段階に至った場合: そこで初めて本格的なデザインシステムの体系化を検討する。
- 原則: **構造は規模に合わせて調整する。** 小さいうちは過剰な体系化を避ける。
