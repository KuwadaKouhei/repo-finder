# Test Philosophy

> 本プロジェクトのテストの拠り所。テストを4つの層に分け、各層が「何を保証するか」を明確にする。
> 具体のテスト項目は REQUIREMENTS.md §3.6 / DETAIL_DESIGN.md §5、全体思想は DESIGN_PHILOSOPHY.md §5 を参照。

---

**カバレッジの数値ではなく「壊れたら困る境界」に確信を置く。** 

---

## 1. テストの4層構造（本プロジェクトの中心戦略）

テストを保証対象の異なる4つの層に分ける。番号が上がるほど検証範囲が広く、実行コストが高い。

| 層 | 何のテストか | **何を保証するため？** | 道具 |
| --- | --- | --- | --- |
| ① ユニットテスト | 純粋関数の単体検証 | **最も壊れやすいロジックを保証するため** | Vitest |
| ② コンポーネントテスト | 単一コンポーネントの props ベース検証 | **ユーザーが見るUIの責務を保証するため** | Vitest + Testing Library |
| ③ 結合テスト | 複数部品＋通信境界を繋いだ検証 | **個々の正しさではなく、繋いだときの正しさ（部品同士の協調）を保証するため** | Vitest + Testing Library + MSW |
| ④ E2Eテスト | 実ブラウザでの端から端の検証 | **実際のユーザー体験を保証するため** | Playwright |

> **用語の注記**:
>
> - ③結合テスト＝**インテグレーションテスト**（同義。テスティングトロフィーで「インテグレーション最厚」と呼ばれる層がこれ）。
> - この4層の土台として**静的解析**（TypeScript `strict` の型チェック・ESLint）が常時走っている。コードを実行せず字面の時点でバグを検出する「書いた瞬間に走る最速・最安の検査層」であり、テストとして書く対象ではないため表には含めないが、型で防げるバグはここで根こそぎ防ぎ、①以降をロジックの誤りに集中させる役割を担う（CI でも lint / typecheck を必須にする）。

### 各層の理由（なぜその保証が必要か）

- **① ユニット**: アダプタのような変換ロジックは、API仕様の罠（`watchers_count`）やエッジケース（null・欠損・不正URL）が集中する「最も壊れやすい場所」。ここが壊れると全画面に波及するため、最初に・最も細かく固定する。
- **② コンポーネント**: UIの責務（language が null なら「言語情報なし」を出す、1ページ以下ならページネーションを出さない）は、ロジックが正しくても表示が間違っていればユーザーには壊れて見える。表示の契約を部品単位で保証する。
- **③ 結合**: ①②がすべて通っても、「検索したら一覧が出る」「新規検索で page がリセットされる」という**部品をまたぐ振る舞い**は誰も保証していない。バグの多くは部品のつなぎ目で起きるため、ここを最も厚くする（トロフィー配分）。
- **④ E2E**: ①〜③は jsdom 上のシミュレーション。本物のブラウザ・本物の Next.js サーバー（RSC含む）で動くことは E2E だけが保証できる。最終的にユーザーが体験するのはこれなので、最後の安全網として最小限（ハッピーパス）を置く。

### 各層の対象（本プロジェクトでの割り当て）

| 層 | 対象 |
| --- | --- |
| ① | アダプタ（変換・null・watchers=subscribers_count・URLスキーム検証）/ format（compact表記） |
| ② | RepositoryCard（表示項目・null表示・フォールバック）/ Pagination(表示条件・無効化・クランプ) / SearchBox（空送信無効・クリア）/ 詳細系コンポーネント |
| ③ | 検索フロー（検索→一覧 / 0件→空 / 403→エラー / pageリセット）/ ソート統合（タスク15後に追記）/ ビュー切替（タスク16後に追記）/ APIクライアントのエラー写像（実装済・①と③の中間） |
| ④ | ハッピーパス1本（検索→結果→詳細遷移→7項目表示）。多層に重ねない |

### 進め方（直列にしない）

- 「①を全部→②を全部→…」とは進めない。**機能実装のたびに ①② を書き、③は機能の節目で追記、④は提出前に1本**。
- これにより、後続の機能追加・リファクタが常にテストに守られた状態を保つ。

## 2. 配分：テスティングトロフィー

本プロジェクトではテスト戦略として**テスティングトロフィーモデル**を採用する。
4層をどの厚さで書くかの配分は**テスティングトロフィー**（Kent C. Dodds）に従う。

- 配分は下から **静的解析 > ①ユニット > ③インテグレーション（最厚）> ④E2E（少）**。トロフィーの形（真ん中が膨らむ）が示す通り、③結合テストを最も厚くする。②コンポーネントテストはユニットとインテグレーションの中間に位置する。
- スローガン: 「テストは書く。多すぎず。ほとんどインテグレーション。」
- **なぜピラミッド（ユニット最厚）でなくトロフィーか**: フロントエンドのバグの多くは部品同士のつなぎ目（操作→状態→表示）で起きる。ユニットを積み上げても「ユーザーの操作が通る」確証になりにくい。ユーザーの使い方に近いインテグレーションテストが、1テストあたりの確信度／コストが最良。
- 土台の静的解析（TypeScript `strict`・ESLint）を広義のテストの一層と捉え、型で防げるバグはコストゼロでここで防ぐ。

---

## 3. 対象：振る舞いをテストする

- **指針**: 「テストはソフトウェアの使われ方に似ているほど確信を与える」（Testing Library guiding principle）。
- 内部state・関数名・クラス名といった**実装詳細を検証しない**。`getByRole` / `getByLabelText` / `getByText` で、ユーザーから見た振る舞いを検証する。`data-testid` は最終手段。
- **なぜか**: 実装詳細に紐づくテストはリファクタしただけで壊れ、変更の足かせになる。振る舞いに紐づけば、内部をどう書き換えても振る舞いが同じなら通り続ける。
- 「テストが壊れやすい＝設計が悪いシグナル」と捉え、テストの壊れやすさを設計改善の手がかりにする。
- 実例: 空白のURLエンコード検証で `%20` という特定表現を期待したテストが、等価な `+` エンコードの正しい実装を落とした。期待値を「デコード後に元へ戻ること」に修正（実装の選択ではなく仕様の本質を検証する）。

## 4. タイミング：部分的TDD（テストファースト）

- 全面的なTDDは課さない。ただし**①ユニット層（ドメイン型とアダプタ）はテストファースト**で書く（先に失敗するテスト→実装）。
- **なぜアダプタか**: 純粋関数で書きやすく、かつ「APIとUIの契約」という壊れたら最も困る境界。先にテストで固定すると、契約が明文化され実装がぶれない。
- ②③は実装後に書く（「振る舞いの固定」が目的。探索的なUI実装にテストファーストは向かない）。
- **テスト容易性を設計の制約として先に効かせる**: 表示コンポーネントは props のみ受け取る純粋な部品にし、データ取得を混ぜない。これにより②が props を渡すだけで書ける。

---

## 5. モック：デトロイト学派寄り（MSWで境界モック）

- できるだけ本物を使い、モックは**ネットワーク境界のみ**に絞る。**MSW** で GitHub API への通信を傍受する。
- MSW は層ではなく**道具**。主戦場は③結合テストだが、APIクライアントの検証にも使う。
- **なぜか**: `fetch` を直接スタブすると実装に結合し、本番と乖離する。境界でモックすれば、実際のクライアントコードをそのまま走らせられ、本番に近いまま検証できる。

### 背景：二つの学派

| | ロンドン学派（mockist） | デトロイト学派（classicist） |
| --- | --- | --- |
| 方針 | テスト対象を**孤立**させ、依存先を全てモックに差し替える | できるだけ**本物**を使い、モックは必要最小限の境界のみ |
| 利点 | 失敗箇所の特定が速い | 本番の動きに近く「本当に動く」確信が高い。リファクタに強い |
| 欠点 | 本物同士を繋いだとき動く保証がない。実装に結合し壊れやすい | 失敗時の原因特定がやや粗い |

- 本プロジェクトは後者寄り。③結合テストでは本物のコンポーネント・アダプタ・クライアントを繋いだまま、ネットワーク境界の一箇所だけを MSW で偽装する。
- モックの呼び出し検証（「この引数で呼ばれた」）は実装詳細の検証なので行わない（§3と同じ価値観）。
- 例外として、②コンポーネントテストで next/navigation（router）等のフレームワーク境界をモックすることは許容する（jsdom に存在しないため）。

---

## 6. 目的：カバレッジを目的にしない

- 網羅率の**数値目標は掲げない**。数値を満たすための価値の低いテスト量産を避ける。
- カバレッジは「テストされていない危険な箇所」を見つける指標としてのみ使う。
- **「壊れたら困る境界」に厚く置く**。本アプリでは ①アダプタ（契約） > ③検索フロー > ②表示の契約 > ④ハッピーパス の順で重要。

---

## 7. テストしないもの（YAGNIをテストにも適用）

- Next のルーティング内部、shadcn/ui・next-themes の挙動そのもの、具体的なCSSクラス名・色・余白（デザインはテストしない）。
- アニメーションの動き自体（reduced-motion 対応の有無は確認するが、動きの検証はしない）。
- E2E の多層化（クロスブラウザ・多シナリオ）。ハッピーパス1本に留める。

---

## 8. RSCのテストに関する注意（現実的事情）

- async な Server Component（page.tsx）は Testing Library で直接 render できない。
- 対策: ③結合テストは「async 部分を await して得た JSX を render する」か「クライアント部品の連携単位」で行い、本物の RSC を通した検証は④E2E に任せる。
- フェッチと表示を分離した設計（§4）が、この制約下でのテスト可能性を支えている。

---

## 9. 生きたドキュメントとしてのテスト

- 具体的なテストケースは仕様書として別途書かず、テストコードの `describe` / `it` の説明文（日本語）に語らせる（二重管理を避ける）。
- 例: `it('watcherには watchers_count ではなく subscribers_count を使う')`。テスト実行結果の一覧がそのまま仕様の一覧になり、実装と乖離しない。

---

## 9.5 テストファイルの配置（同階層方式）

- 対象が明確なテスト（①②）は**対象ファイルの真隣**に置く（`repository-card.tsx` の隣に `repository-card.test.tsx`）。
- **理由**: テストの有無が一目で分かり、**テストのし忘れを防げる**。リネーム・削除時にテストが視界に入り孤児テストを生まない。仕様書（§9）として実装とセットで読める。コロケーションの原則（UI_DESIGN_PHILOSOPHY.md）と一貫。
- `__tests__/` への集約は不採用（実装フォルダの一覧性より、テスト有無の可視性と対応関係の維持を優先。配置規則を全層で1つに統一）。
- 例外: 部品をまたぐ③結合テストは機能フォルダ直下、テスト基盤は `src/test/`、④E2E は導入時に専用フォルダ（`e2e/`）。

---

## 10. ツール

- **Vitest**（ランナー・①②③）/ **React Testing Library**（②③の振る舞い検証）/ **MSW**（③の境界モック）/ **Playwright**（④・提出前に導入）。
- 実行環境は jsdom（①〜③）。CI で lint・typecheck・test を必須にする。
